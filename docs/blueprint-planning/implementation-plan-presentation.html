UX consistency Existing design language

The implementation should feel dense, calm, and operational.

Cadenza is an operator tool, not a marketing site. New modules should reuse the existing shell patterns and avoid ornamental layouts, oversized heroes, and one-off component language.

Visual language

  • Warm paper workspace with espresso sidebar.
  • Bordeaux for brand emphasis, navy for action and selected state.
  • 8px-radius cards, panels, drawers, and repeated items.
  • Compact headers and segmented controls for views and modes.

Interaction language

  • Icon-led controls for tools, search, filters, route actions.
  • Tabs for record sections, menus for option sets, toggles for binary state.
  • Dense list-detail layouts for scanning and repeated work.
  • Mobile scope declared per workflow, not inherited accidentally.

Product language

  • Every page shows real work first: queues, records, histories, status.
  • Empty states explain what is absent and provide the next valid action.
  • Blocked sections stay visibly gated in planning, not half-built in UI.
  • Reports and dashboards expose only authorized source data.
Design bar: every module should be usable by an admin on a large operator display, readable in Hebrew/RTL, and stable at 390x844 for workflows that have a real mobile user.

Operating model Board + product + engineering

Each release slice should answer three audiences at once.

Board view

  • Which business capability becomes usable?
  • Which risks are retired or intentionally deferred?
  • Which blocked decisions still constrain the roadmap?

All-hands view

  • What changes for operators in the app shell?
  • Where should teams look for the new workflow?
  • How does it stay consistent with current Cadenza behavior?

Developer view

  • Which packet section is in scope?
  • Which RLS, mapping, smoke, RTL, and backfill gates are required?
  • Which adapter and route policies must not be bypassed?
