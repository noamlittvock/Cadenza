## 01 CANONICAL SUMMARY

Cadenza Management Platform is a calendar-first conservatory management web application for internal admin use. The existing application manages staff scheduling, room booking, and financial reporting via a live calendar. This specification defines a data model migration — replacing hardcoded category schemas with a dynamic Firestore-native Activity registry — and adds two new first-class modules: a Student module with full profiles and pedagogical records, and a Teacher Hours Reporting system that pre-populates from calendar events for event-by-event confirmation, correction, or absence marking, with a line-by-line admin reconciliation view. It also introduces Calendar Subscriptions, allowing filtered iCal feeds to be generated and shared externally. A mandatory Phase 10 — RTL & Translation QA — ensures all modules are fully translated and RTL-verified before release. The existing UI is not redesigned.

## 06 PHASE DEFINITIONS

> **MIGRATION PRINCIPLE:** Each phase must leave the existing app fully functional at its end. No phase should break live features. UI changes are only permitted where forced by data model changes confirmed in this spec.

### Phase 1 — Activity Registry (Foundation)

Create the activities Firestore collection. Build the Activities and Subcategories tabs in the Config Hub. Implement archive pattern. Build inline subcategory creation for use in later phases.

**Definition of Done:** Admin can create, edit, and archive Activities and Subcategories from Config Hub. Archived items hidden from dropdowns, retained in historical records. Inline subcategory creation component built and ready for integration. Existing app behaviour unchanged.

### Phase 2 — Staff Member Migration

Expand the existing teachers Firestore collection with new fields: Position Titles, Teaching Assignments (Activity + Subcategory with date range, effective dating, and ensemble flag), credentials, bio, employment type, date of joining, isArchived. Build the Staff Members module as a new top-level sidebar item. Remove Teachers tab from Manage hub (now in Staff Members sidebar). Implement archive flow with Admin Inbox task trigger for unassigned students.

Teaching Assignments are **effective-dated**: any edit to a Teaching Assignment triggers an effective date prompt (system suggests today, admin can change). The system creates a new assignment record from the effective date forward; all CalendarEvents before the effective date retain a snapshot of the old assignment rate.

Position Name in Add/Edit Staff Member modal is a dropdown sourced from the Positions list in Config Hub, with an inline "Add Position" option to create a new position without leaving the modal. Rate Type on each Position Assignment must offer all three values: Hourly, Global Monthly, Per Event. VAT Type, VAT Value, Overhead, and Social fields on Position Assignments are retained unchanged.

**Definition of Done:** Full Staff Member CRUD in new sidebar module. All existing teacher data intact and readable. New fields available but not required for existing records. Archive triggers Admin Inbox task. Financial Dashboard still references staff correctly.

### Phase 3 — CalendarEvent Migration (Dual-Field Transition)

Add activityId and subcategoryId fields to CalendarEvent schema alongside existing categoryId and classification. Update event creation and edit flows to write new fields and read from Activity registry for dropdowns. Old events continue to read categoryId and classification. Both field sets coexist.

**Event modal changes required by this migration:**
1. Category dropdown → Activity dropdown sourced from Activity registry. Hardcoded groups (Teacher Lessons, General Events) removed; replaced by flat dynamic list of Activities.
2. Subcategory cascading dropdown added beneath Activity — appears only after Activity is selected, populates with that Activity's Subcategories.
3. Teacher dropdown filters to show only Staff Members who have a Teaching Assignment matching the selected Activity — three-level cascade: Activity → Staff Member (filtered by Activity) → Position (filtered by Staff Member).

eventIntent is derived from Activity.type (INSTRUCTIONAL or OPERATIONAL) and written automatically on event creation — no separate field in the modal. staffMemberIds (string[]) is written on every new event with the selected staff member at index 0.

**Definition of Done:** New events created with activityId and subcategoryId. Existing events unmodified and still render correctly. Calendar UI unchanged. Financial Dashboard and filters continue to function using both field sets. No data loss.

### Phase 4 — Student Module

Create students Firestore collection. Build Student module as new top-level sidebar item. Implement full Student profile: identity, minor/adult detection, grade auto-increment, guardian contacts (repeatable), Student Assignments (Activity + Subcategory + Staff Member filtered), Ensemble roster membership, Pedagogical Record (lesson history as live event query, manual recital history and report cards), notes, documents. Implement archive pattern.

**Definition of Done:** Full Student CRUD. Student Assignments filter Staff Members by matching Teaching Assignment. Ensemble roster add/remove works. Pedagogical Record surfaces linked calendar events and accepts manual entries. Student archiving works cleanly.

### Phase 5 — Teacher Google Calendar Sync (Per Staff Member)

Add per-Staff-Member Google Calendar sync settings to Staff Member profile. Admin enables sync, enters teacher's Google Calendar ID. App pushes that staff member's events to their personal Google Calendar — one-way only. No events flow back into the app from teacher calendars.

**Definition of Done:** Admin can enable/disable Google sync per Staff Member. Events assigned to that Staff Member push to their Google Calendar on creation and deletion. Existing admin-level two-way sync unaffected.

### Phase 6 — Calendar Subscriptions

Build iCal feed generation. Admin can create a named subscription with any filter combination (teacher, tag, position, room, activity). System generates a unique tokenized iCal URL. Admin can view, copy, and revoke subscription links. Feed updates automatically as calendar events change. Manage subscriptions from Config Hub or Settings (placement TBD — see §14).

**Definition of Done:** Admin can generate, share, and revoke iCal subscription URLs. Feeds are filterable by at least: teacher, tag, position, room, activity. Feed renders correctly when subscribed in Google Calendar. Revoked links return no data.

### Phase 7 — Teacher Hours Reporting

Build tokenized Hours Report form (standalone page, no login). The form pre-populates with the teacher's scheduled calendar events for the reporting period. For each event, the teacher responds: confirmed as scheduled, different duration (with correction), or not completed (with reason drawn from the admin-managed absence reasons list in Config Hub). The teacher can also add manual entries not on the calendar, with an optional Activity + Subcategory selector and optional freeform description.

Admin generates the link per Staff Member per reporting period from the Staff Member profile and sends it to the teacher. Build Hours Comparison View in Financial Dashboard area showing reported hours vs calendar event hours as a line-by-line reconciliation (not just totals). Admin can make adjustments or flag discrepancies.

**Definition of Done:** Admin can generate and send tokenized form link from Staff Member profile. Teacher can submit hours without logging in. Form pre-populates with calendar events for the period; teacher can confirm, adjust, or mark not completed against each event, selecting absence reasons from the admin-managed list. Teacher can add manual entries with optional Activity + Subcategory and freeform description. Admin sees line-by-line reconciliation in Financial Dashboard area. Comparison correctly maps submitted entries against calendar events for the same period and staff member.

### Phase 8 — Admin Inbox

Build shared Admin Inbox accessible to all admin-tier users. Notifications feed (informational, timestamped) and Task list (actionable, markable as done). Implement all confirmed task triggers. Implement notification triggers as identified during build.

**Definition of Done:** Admin Inbox visible to all admin-tier users showing shared state. Tasks markable as done. At minimum: Staff Member archived with active assignments generates unassigned student task automatically.

### Phase 9 — CalendarEvent Cleanup Migration

Once all features are stable and all new events use activityId + subcategoryId, run a one-time cleanup script to backfill activityId and subcategoryId on all old events by mapping their categoryId and classification values to Activity registry entries. After validation, the old fields (categoryId, classification) can be deprecated.

**Definition of Done:** All CalendarEvents have activityId and subcategoryId populated. Old fields retained as legacy but no longer read by the application. Validated against Financial Dashboard and Calendar View with zero data discrepancies.

### Phase 10 — RTL & Translation QA

Conduct a full RTL and translation quality assurance pass across all modules, including every new module introduced in Phases 1–9. Validate that all new UI strings have been added to the existing translations Firestore collection from day one. Run AI programmatic validation: confirm all translation keys are present, no strings are hardcoded in UI code, and RTL CSS is consistent throughout. Conduct owner manual walkthrough of the entire application in Hebrew.

**Definition of Done:** (1) AI programmatic validation passes — all translation keys present in Firestore translations collection, zero hardcoded UI strings, RTL CSS consistent across all views. (2) Owner manual walkthrough in Hebrew completed with no unresolved layout or translation issues. **Both conditions must be satisfied; either alone is insufficient.**

## 07 PRODUCT DEFINITION

### What It Is
A calendar-first conservatory management platform for internal admin use. The calendar is the operational engine — every lesson, ensemble rehearsal, recital, and administrative event is logged here. Financial reporting, student history, and hours reconciliation all derive from calendar events. Staff profiles, student profiles, rooms, and activities are the supporting entities that make events meaningful.

### What It Is Not
Not a parent or student portal. Not a payment or billing system. Not a public-facing website. Not a mobile app (v1). Not a skill assessment platform (v1). Not a UI redesign — the existing interface is preserved.

### Core User Flow — Adding a Staff Member
1. Admin navigates to Staff Members (sidebar) → Add New.
2. Fills identity: full name, DOB, date of joining, governmental ID, employment type.
3. Adds Position Titles (repeatable): selects from managed list.
4. Adds Teaching Assignments (repeatable): Activity → Subcategory cascades → select or add inline → date range → if Ensemble, roster enabled.
5. Optionally enables Google Calendar sync: toggles on, enters teacher's Google Calendar ID.
6. Adds credentials, bio, notes, documents as needed. Saves.

### Core User Flow — Adding a Student
1. Admin navigates to Students (sidebar) → Add New.
2. Fills identity: full name, DOB (age + minor status auto-calculated), grade (if minor), governmental ID.
3. If minor: adds guardian contacts (repeatable).
4. Adds Student Assignments (repeatable): Activity → Subcategory → system filters Staff Members by matching Teaching Assignment → admin selects → if Ensemble, student added to roster.
5. Adds notes, documents. Saves.

### Core User Flow — Generating a Calendar Subscription
1. Admin navigates to Manage → Calendar Subscriptions tab.
2. Creates new subscription: names it, sets filters (any of: teacher, tag, position, room, activity).
3. System generates a unique tokenized iCal URL.
4. Admin copies link and shares with relevant person.
5. Recipient subscribes in their own Google Calendar or iCal-compatible app.
6. Feed updates automatically. Admin can revoke at any time.

### Core User Flow — Teacher Hours Reporting
1. Admin navigates to Staff Member profile → generates Hours Report link for a specific period.
2. Admin sends link to teacher (outside the app).
3. Teacher opens tokenized URL, sees their reporting period with calendar events pre-populated. For each event: confirmed / different duration (enters correction) / not completed (selects absence reason from managed list). Adds any additional manual entries with optional Activity + Subcategory and optional freeform description. Submits.
4. Admin navigates to Financial Dashboard → Hours Comparison View.
5. Sees line-by-line reconciliation: each calendar event alongside the teacher's response, for that staff member and period.
6. Makes adjustments or flags discrepancies directly from the comparison view.

## 09 EDGE CASES & FAILURE HANDLING

| Scenario | Required Behaviour |
|---|---|
| Staff Member archived with active Student Assignments | Archive completes immediately. Admin Inbox task generated listing all affected unassigned students. |
| Subcategory added inline doesn't exist in master list | Confirmation prompt shown before committing to master list. Prevents typos becoming permanent entries. |
| Student turns 18 while enrolled as minor | Grade and guardian fields remain. Admin manually updates profile. No auto-conversion in v1. |
| Two Staff Members hold the same Teaching Assignment | Both appear as valid options during student assignment. Admin makes explicit selection. |
| Ensemble Activity exists but has no active Staff Member assignment | Does not appear in cascading dropdowns. Visible in Config Hub as active but unassigned. |
| Old CalendarEvent read during dual-field transition | App reads categoryId and classification fields. Renders correctly. No migration required at read time. |
| New CalendarEvent created during dual-field transition | App writes activityId and subcategoryId. Does not write categoryId or classification. |
| Teacher submits Hours Report after admin has already adjusted | Submission is timestamped. Comparison view shows both versions with timestamps. Admin decides which to use. |
| Calendar Subscription link is revoked | iCal feed returns empty or 404. Subscriber's calendar app stops receiving updates. |
| Teacher Google Calendar sync fails | Event saved in app. Sync error surfaced to admin on Staff Member profile or Admin Inbox. Manual retry available. |
| Admin Google Calendar sync fails (existing behaviour) | Retain existing error handling. No changes. |
| School Year Rollover Date reached — student has no grade set | Grade field remains blank. No auto-increment on null. Admin prompted via Admin Inbox notification. |
| Activity or Subcategory archived while referenced by active assignments | Archived item hidden from new selections. All existing records retain the reference intact. |
| Cleanup migration script runs — unmapped categoryId found | Script logs unmapped events. Does not auto-assign. Admin reviews unmapped list before finalising cleanup. |

## 12 EVENT STATUS & BLACKOUT BEHAVIOR

CalendarEvents have four mutually exclusive status values. All code, queries, and UI references must use these exact values.

| Status | Meaning | Counts Toward Hours | Notes |
|---|---|---|---|
| ACTIVE | Scheduled and live | Yes | Normal state |
| CANCELED | Explicitly canceled by admin | No (counts as Cancel #) | Existing behavior retained |
| PARKED | Suppressed by a blackout period | No | Auto-set when blackout is applied to range. Hidden from Hours Report form. Any admin can unpark. Event data fully intact. |
| ARCHIVED | Permanently removed from active views | No | Data retained for historical queries. Not reversible. |

**Blackout mechanics:** When admin applies a blackout to a date range, all ACTIVE events within that range are automatically set to PARKED. CANCELED events within the range are unaffected. PARKED events are:
- Hidden from calendar view (controlled by existing Hide Blackouts toggle)
- Excluded from Financial Dashboard hour calculations
- Hidden from the Teacher Hours Report form

Any admin can unpark events. Unparking restores status to ACTIVE.

The Hide Canceled toggle does **NOT** hide PARKED events — the Hide Blackouts toggle controls PARKED visibility.

## 13 DEVELOPER TOOLS TEST SUITE

A built-in test suite lives in Super Admin → Developer Tools. Each test shows: test name, description, last run timestamp, pass/fail status, and a Run button. Tests can be run individually or as a full suite. Results show a summary count and expandable detail per test.

Creates synthetic data and validates the system processes it correctly. Cleans up after itself unless Keep Data is toggled.

**G3 — Event Creation — All Activity Types.** Creates one event of each Activity type. Verifies each event writes activityId, subcategoryId, eventIntent, and staffMemberIds correctly, appears on calendar, and is filterable by Activity and Staff Member via three-level cascade.

**G4 — Recurring Event Series.** Creates a weekly recurring event for 8 weeks. Verifies all 8 instances are created with correct dates, all carry the same activityId/subcategoryId/staffMemberIds, and that editing one instance vs all instances behaves correctly.

**G5 — Financial Dashboard Population.** Creates events across three Staff Members with different rate types (Hourly, Global Monthly, Per Event). Verifies all three appear as separate line items, totals calculate correctly per rate type, and the Per Event summary card shows the correct value.

**G6 — Hours Report Full Flow.** Generates a tokenized Hours Report link. Simulates a submission with two confirmed events, one adjusted duration, one not-completed with absence reason, and one manual entry with Activity + Subcategory. **G7 — Calendar Subscription.** Creates a subscription filtered by one Staff Member and one Activity. Verifies iCal feed URL is generated, returns valid RFC 5545 content, contains only matching events, and returns empty after revocation.

**G9 — Activity Registry Inline Add.** During event creation, adds a new Subcategory inline. Verifies the Subcategory is persisted to the Activity registry, appears in future dropdowns, and the event is saved with the correct subcategoryId.

**G10 — Archive Cascade.** Archives a Staff Member with two active Student Assignments. Verifies Staff Member is hidden from active views, both Student Assignments are flagged, and Admin Inbox receives exactly one task listing both affected students.

### Tier 2 — Scenario Tests
Edge cases and boundary conditions. Uses minimal synthetic data. Disabled on production orgs.

**S1 — Dual-Field Transition Read.** Creates a legacy event with only categoryId + classification (no activityId/subcategoryId). Verifies Calendar View renders it correctly, Financial Dashboard counts it correctly, and no errors are thrown.
