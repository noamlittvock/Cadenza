## Risks and regressions

- **Iteration 1 — fallback chain on event title:** `(e as any).title || (e as any).activity || e.name` works under TS but loses compile-time safety. If the schema renames `name` again, the search will silently stop matching events.
- **Iteration 2 — duplicated conflict computation:** the new `unresolvedConflictCount` memo recomputes `detectRoomConflicts` independently of the existing `conflictingIds` (one is system-wide, one filtered). Performance fine on current densities, but a follow-up should merge them.
- **Iteration 2 — pill label prefix risk in he-IL:** active filter pill labels are derived by stripping "All " from existing `cal.filter.*_all` keys. Hebrew strings may not start with literal "All ", so pill labels may include the original prefix. Cosmetic; per-noun keys queued for next run.
- **Iteration 4 — bulk archive still uses `window.confirm`:** out of scope this run. Worth a single follow-up iteration to migrate to the new Modal so the surface is consistent.
- **ESLint gate unreachable:** every worker passed tsc + visual but the lint side of the orchestrator's verification gate could not run. A minimal `eslint.config.js` should land before the next Branch Lab run.
- **Dev server liveness during parallel dispatch:** my dev server stopped at some point during Worker C's run (curl returned 000); restarted before iteration verification. Cause unclear. Future runs should monitor and auto-restart, or have workers spawn their own ephemeral preview build.

## Operational lessons

- **i18n pre-stage in constants.ts works** — there is no `src/lib/i18n.ts` in this codebase; pre-staging directly into the existing TRANSLATIONS object with a unique `bl01_*` prefix gave workers safe, race-free access to all keys without anyone writing to constants.ts mid-run.
- **ESLint v9 flat-config gap is a project-wide issue, not a Branch Lab issue** — the orchestrator's lint gate is unreachable until the project ships `eslint.config.js`. Future Branch Lab runs should detect missing flat config in Phase 1 and either fail loud, scaffold a minimal config, or document the gap as it did this run.
- **2429-line single-file workers can ship coherent multi-move iterations** — Worker B successfully bundled three coherent moves (filter pills + conflict badge + focus ring) into 133 net lines without refactoring the host file. The "single coherent worker" pattern from the orchestrator §6 paid off.
- **Dev server liveness is fragile across parallel worker dispatch** — at least one worker found 3001 occupied and started its own server on 3002; mine apparently stopped during the run. Future runs should treat the dev server as a managed resource (launch in a session-persistent process, monitor, restart).
- **Audience description in run config can lie about the product** — the input said "musicians and creators" but Cadenza is admin-facing; the orchestrator should always reconcile against actual product evidence (README, types, primary surfaces) and document the divergence in product-understanding.md before sliding into research.
- **Workers can simplify beyond spec when honest about it** — Worker D's reuse of `archiveCascade` state instead of duplicating with a new `archiveConfirmTarget` is a cleaner outcome than the spec asked for, and the worker disclosed it. That's the kind of "scope evolution" the lab wants to encourage.
